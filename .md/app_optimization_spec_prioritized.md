# 图片笔记 App 优化文档（按优先级）

## 1. 文档目标

本次优化只围绕以下核心定位展开：

- 这是一个 **图片驱动的笔记 App**。
- 核心能力只有三件事：**记录、打标签、检索**。
- **不做训练功能**、不做决策打分、不做案例刷题、不做错题训练流。

本次优化目标是让现有 App 更适合日常复盘记录，重点提升：

1. 记录效率
2. 标签灵活性
3. 检索能力
4. 数据可维护性
5. 未来可扩展性

---

## 2. 产品边界

### 必做范围

- 图片导入
- 笔记创建与编辑
- 自定义标签
- 单张图片支持多个标签
- 多条件筛选与搜索
- 收藏
- 最近使用项
- 本地存储
- 本地备份与恢复（低优先级）

### 明确不做

- 训练模式
- 看图判断涨跌/做多做空
- 遮挡答案再揭晓
- 错题训练流
- 交易评分系统
- 复杂统计分析平台
- 社交功能
- 云同步（当前阶段不做）

---

## 3. 优先级总览

## P0（必须先做）

这些改动直接影响产品是否好用，必须优先落地。

### P0-1. 标签系统改造为“完全自定义 + 多标签”

#### 目标

解决当前标签能力过弱的问题，让用户可以自由组织图片笔记。

#### 要求

- 用户可以创建自定义标签
- 用户可以编辑标签名称
- 用户可以删除标签
- 单张图片笔记可以绑定多个标签
- 标签支持颜色或图标（可选，若实现成本高可先不做）
- 创建/编辑笔记时，可以快速选择多个标签
- 支持展示该笔记的所有标签
- 支持根据一个或多个标签筛选笔记

#### 设计要求

- 标签不能只保留单个 `type_tag_id`
- 必须改为 **Note 与 Tag 多对多关系**
- 标签是统一资源，不是写死枚举
- 标签不区分“系统标签/用户标签”，统一按自定义标签处理

#### 验收标准

- 用户新建一条笔记时，可以选择多个标签
- 用户编辑笔记时，可以新增/移除已有标签
- 列表页和详情页能正确展示多个标签
- 筛选页可以按多个标签联合过滤

---

### P0-2. 优化新建笔记流程，降低记录成本

#### 目标

让“导入图片并记一条笔记”的操作足够快。

#### 要求

- 新建笔记流程尽量缩短
- 进入创建页后，优先展示：
  - 图片预览
  - 标题/备注
  - 标签选择
  - 品种
  - 周期
  - 时间
  - 收藏开关
- 默认带出最近使用的字段值：
  - 最近使用标签
  - 最近使用品种
  - 最近使用周期
- 支持快速连续创建多条笔记
- 图片选中后尽快进入编辑页，不增加多余步骤

#### 设计要求

- 表单字段顺序以“最常用优先”为原则
- 减少用户重复输入
- `RecentUsages` 要真正参与创建流程，而不是仅存储不用

#### 验收标准

- 连续添加多张图片时，用户无需重复选择高频字段
- 新建笔记时默认值可自动回填
- 整体创建路径明显短于当前版本

---

### P0-3. 检索与筛选能力升级

#### 目标

让用户能快速从大量图片笔记中找到想看的内容。

#### 要求

- 支持关键词搜索：标题、备注
- 支持多标签筛选
- 支持品种筛选
- 支持周期筛选
- 支持时间范围筛选
- 支持仅看收藏
- 支持多个条件组合筛选
- 支持保存常用筛选条件

#### 设计要求

- 筛选条件要能组合，不是互斥
- Saved Filters 要产品化，不只是底层数据结构
- 搜索结果页要清晰显示当前已生效的筛选条件

#### 验收标准

- 用户能保存一个常用筛选方案并反复使用
- 用户可以同时按“多个标签 + 品种 + 周期 + 收藏”进行筛选
- 搜索结果与筛选结果正确、稳定

---

### P0-4. 数据模型重构，保证后续不会越做越乱

#### 目标

修正当前模型中不利于后续维护的设计。

#### 要求

- 将笔记与标签关系改为多对多
- 对高频字段做规范化处理，避免脏数据：
  - symbol
  - timeframe
- 增加 `updated_at`
- 增加软删除字段 `deleted_at`（可选但推荐）
- 增加 `is_favorite`
- 保留 `created_at`
- 图片路径字段需要稳定、可迁移

#### 建议表结构

### notes

- id
- image_path
- title
- content
- symbol
- timeframe
- trade_time
- is_favorite
- created_at
- updated_at
- deleted_at

### tags

- id
- name
- color （可选）
- icon （可选）
- created_at
- updated_at

### note_tags

- note_id
- tag_id

### saved_filters

- id
- name
- filter_payload_json
- created_at
- updated_at

### recent_usages

- id
- field_name
- field_value
- updated_at

#### 验收标准

- 一个 note 能关联多个 tag
- 一个 tag 能被多个 note 复用
- 删除 tag 后，关联关系能被正确清理
- 筛选和列表查询性能正常

---

## P1（重要，但可以在 P0 后做）

这些优化不会决定 App 能不能用，但会显著提升体验。

### P1-1. 列表页与详情页展示优化

#### 目标

让浏览体验更清晰，信息密度更合理。

#### 要求

- 笔记卡片展示：
  - 缩略图
  - 标题或备注摘要
  - 标签
  - 品种/周期
  - 创建时间
  - 收藏状态
- 详情页展示完整大图与完整信息
- 标签区域可点击，点击后直接进入对应筛选结果

#### 验收标准

- 列表页可快速识别每条笔记的核心信息
- 详情页能查看全部标签和全部备注
- 点击标签能触发检索或筛选

---

### P1-2. 标签管理页完善

#### 目标

让标签真正可维护，而不是越用越乱。

#### 要求

- 单独提供标签管理页
- 支持新增标签
- 支持编辑标签名称
- 支持删除标签
- 展示每个标签被使用的次数
- 支持按名称排序 / 按使用次数排序

#### 验收标准

- 用户可以集中整理标签
- 用户可以清理无用标签
- 标签使用次数展示准确

---

### P1-3. 导入体验增强

#### 目标

继续缩短记录路径。

#### 要求

- 支持多图导入
- 支持导入后逐张快速补充信息
- 允许从相册快速连续选择
- Android 端预留系统分享导入能力（如实现成本合适可纳入本期，否则留接口）

#### 验收标准

- 用户可以一次导入多张图
- 用户能逐张快速完成标记，而不必每次重新进入首页

---

### P1-4. 基础统计信息（仅轻量级）

#### 目标

给用户提供最基础的整理反馈，但不做训练和复杂分析。

#### 要求

只做轻量统计：

- 笔记总数
- 收藏数量
- 标签数量
- 最近 7 天新增数量
- 某标签下笔记数量

#### 明确限制

- 不做交易胜率分析
- 不做训练统计
- 不做策略评估

#### 验收标准

- 首页或设置页可以展示轻量级数量统计
- 所有统计都基于本地数据即时计算或缓存

---

## P2（可选优化）

这些可以放在后续版本，不影响 MVP 可用性。

### P2-1. 本地备份与恢复

#### 目标

防止换设备或误删导致数据丢失。

#### 要求

- 导出本地数据库或 JSON/ZIP 包
- 导入备份文件恢复数据
- 恢复后图片与标签关系正确

#### 验收标准

- 用户可成功导出一份完整备份
- 恢复后笔记、标签、收藏、筛选器均完整

---

### P2-2. 删除回收与批量操作

#### 目标

提升长期使用时的数据管理能力。

#### 要求

- 批量删除笔记
- 批量添加标签
- 批量取消收藏
- 回收站（如果实现 soft delete）

#### 验收标准

- 用户可在列表中进入多选模式
- 多选操作稳定可用

---

### P2-3. 性能与体验打磨

#### 要求

- 大量图片下列表滚动流畅
- 缩略图加载有缓存
- 搜索与筛选响应速度稳定
- 页面跳转和表单保存无明显卡顿

---

## 4. 对 AI 编译器的实现要求

### 架构要求

- 按 feature 组织代码，不要过度设计
- 先保证功能可用与结构清晰，再考虑抽象
- 不要为了“架构好看”制造大量样板代码

### 状态管理

- 沿用现有 Riverpod
- Provider 命名清晰
- 表单状态与列表筛选状态分离

### 数据层要求

- 使用 Drift 实现多对多表关系
- 查询逻辑不要散落在 UI 层
- 搜索、筛选、保存筛选方案都要走 repository/usecase

### UI/交互要求

- 创建流程优先快
- 标签选择支持多选
- 筛选页支持组合条件
- 常用操作尽量减少点击次数

---

## 5. 推荐开发顺序

1. 重构数据库表结构（Tag 多对多）
2. 改造笔记创建/编辑页，支持多标签与最近使用
3. 改造筛选页与搜索逻辑，支持多标签组合筛选
4. 改造列表页与详情页，正确展示标签与筛选状态
5. 增加标签管理页
6. 增加多图导入与快速录入
7. 增加轻量统计
8. 最后再做备份恢复、批量操作、性能优化

---

## 6. 最终验收口径

本轮优化完成后，App 必须满足以下结果：

- 用户可以创建任意自定义标签
- 单张图片笔记可以绑定多个标签
- 用户可以基于多个标签与其他条件进行组合检索
- 新建笔记流程明显更快，能复用最近使用项
- 标签可集中管理，长期使用不会失控
- 整个产品仍然保持“图片笔记 + 检索工具”的定位，不引入训练功能
