# 图片主导交易复盘 App 产品需求文档（PRD）

## 1. 产品概述

### 1.1 产品名称（暂定）
- 中文：图流复盘
- 英文：ChartFlow / SnapReplay / TradeSwipe

### 1.2 产品定位
一款**以图片为主导**的移动端复盘工具，面向期货/交易用户。用户通过上传 K 线截图或盘面图片，快速为图片打上标签（type、时间、品种、级别、结果等），并以类似抖音/小红书的信息流方式回看、筛选和训练自己的交易样本。

它不是传统文档笔记，也不是便利签，而是一个：

**图片驱动的交易案例流系统**。

### 1.3 一句话描述
> 一张图就是一条复盘笔记；记完就能刷，刷久了就形成模式识别能力。

---

## 2. 背景与问题

### 2.1 当前市场问题
当前手机端笔记产品高度同质化，主要分为两类：

1. **文档型笔记**（如 Notion、语雀）
   - 偏重
   - 适合长文档整理
   - 不适合高频快速记录截图型复盘内容
   - 图片只是附件，不是核心对象

2. **轻量便签型笔记**（如 Keep 类）
   - 轻量但结构弱
   - 缺少对图片案例的系统化管理
   - 无法高效按交易模型、品种、时间、结果进行流式回看

### 2.2 用户核心痛点
对于期货复盘用户，尤其是依赖图形样本训练的用户，存在以下问题：

- 复盘主要内容是截图，而不是长文字
- 现有笔记软件录入链路太长
- 图片太多后，很难回看和筛选
- 无法像刷内容流一样高频回顾自己的案例
- 无法高效沉淀“某个 type 的全部样本”
- 样本积累后无法形成训练闭环

### 2.3 本产品解决的问题
本产品核心解决：

- **如何快速记录一张交易截图**
- **如何让图片天然成为笔记主对象**
- **如何让用户像刷短视频一样回看自己的复盘样本**
- **如何通过标签和筛选形成交易模式识别训练**

## 3. 目标用户

### 3.1 核心用户
- 期货交易者
- 股票/期权/外汇等技术分析型交易者
- 高频做复盘、依赖截图积累样本的用户

### 3.2 用户特征
- 经常截 K 线图
- 有自己的一套交易语言或形态标签
- 需要按品种、周期、做多做空、盈亏结果等维度回看
- 希望通过反复刷样本强化盘感和模式识别
- 不喜欢复杂文档型工具

### 3.3 典型用户画像
- 用户每天交易后会截图 5~20 张盘面
- 每张图只需要记录极少量文字
- 最重视的是“后续快速回看和筛选”，而不是“当下写很多”

---

## 4. 产品目标

### 4.1 MVP 目标
在第一版中，实现以下核心闭环：

1. 用户可以快速导入图片
2. 用户可以给图片打标签并保存
3. 用户可以按指定标签流式浏览图片笔记
4. 用户可以按时间、type、品种等条件进行筛选

### 4.2 中期目标
- 建立交易样本库
- 支持训练模式
- 支持 AI 自动识别图片内容并给出初始标签建议

### 4.3 长期目标
从“图片笔记工具”演化为“案例训练系统”，帮助交易者建立自己的模式数据库与识别能力。

---

## 5. 核心理念

### 5.1 产品原则
1. **图片第一，文字第二**
2. **记录尽可能轻，回看尽可能爽**
3. **单条笔记像一张内容卡，而不是一篇文档**
4. **默认服务复盘和训练，不服务长文写作**

### 5.2 核心交互理念
- 用户不是“写笔记”
- 用户是在“存案例”
- 用户不是“翻文件夹”
- 用户是在“刷样本流”

## 6. 功能范围

## 6.1 MVP 功能

### 6.1.1 图片导入
支持以下方式：
- 从相册选择图片
- 从系统分享面板分享到 App
- 拍照上传（低优先级，可后置）

### 6.1.2 创建图片笔记
用户选择图片后，创建一条笔记，字段包括：

#### 必填字段
- 图片（image）
- type（交易类型/形态标签）

#### 推荐字段
- 时间（交易时间/复盘时间）
- 品种（如 FG / RB / M）
- 周期（5m / 15m / 30m / 60m）
- 方向（多 / 空 / 观察）
- 结果（盈利 / 亏损 / 观察 / 错过）
- 备注（一句话总结）

#### 系统字段
- id
- 创建时间
- 更新时间
- 是否收藏
- 是否归档

### 6.1.3 标签体系
支持用户自定义以下标签：
- type 标签
- 品种标签
- 周期标签
- 结果标签

其中 type 为最核心标签，决定刷流入口。

### 6.1.4 流式浏览
支持类似短视频的信息流浏览：
- 单张图全屏展示
- 上下滑切换下一条/上一条
- 底部显示核心字段
- 支持当前筛选条件下的连续浏览

### 6.1.5 筛选与分类
支持按以下条件筛选：
- type
- 品种
- 时间范围
- 周期
- 方向
- 结果
- 收藏状态

### 6.1.6 收藏与重点样本
用户可将高价值案例加入收藏，用于后续重点回看。

### 6.1.7 搜索
MVP 支持基础搜索：
- 搜索备注文本
- 搜索 type 名称
- 搜索品种

---

## 6.2 非 MVP，后续版本功能

### 6.2.1 AI 标签建议
通过 OCR 和图像识别，对导入图片进行：
- 自动识别时间文字
- 自动识别品种名称
- 自动识别周期
- 自动生成初始摘要

### 6.2.2 训练模式
- 随机出图
- 用户先判断 type / 多空 / 是否可做
- 再查看自己历史标注答案

### 6.2.3 统计页
- 某个 type 的总样本数
- 某个品种的盈利/亏损比例
- 不同周期样本分布
- 近期高频错误类型

### 6.2.4 多图关联
允许用户将多张图组成一个案例链：
- 入场图
- 加仓图
- 止盈图
- 止损图
- 复盘总结图

### 6.2.5 云同步
- 多设备同步
- 备份与恢复

## 7. 信息架构

## 7.1 一级页面

### A. 首页
功能：快速进入最近刷流、最近新增、快速创建

模块建议：
- 继续刷样本
- 快速导入图片
- 最近新增
- 常用筛选（如“摸顶难”“RB 30m”“亏损案例”）

### B. 发现/流页面
功能：主浏览页面，像抖音一样刷图

### C. 创建页面
功能：导入并标注图片笔记

### D. 筛选页
功能：组合筛选条件，进入指定样本流

### E. 我的/设置页
功能：管理标签、导出数据、设置主题等

---

## 8. 核心页面设计

## 8.1 首页

### 页面目标
让用户在 3 秒内知道下一步做什么：
- 继续刷
- 录入新图
- 进入某个专题流

### 页面模块
1. 顶部标题 + 搜索入口
2. “继续刷样本”入口
3. “导入图片”主按钮
4. 常用 type 快捷入口
5. 最近新增样本列表
6. 收藏样本入口

### 交互要求
- 首屏突出“导入图片”和“继续刷”
- 不出现传统文档列表式压迫感

---

## 8.2 创建笔记页

### 页面目标
让用户在 **5~10 秒内完成一条图片笔记**

### 录入流程
1. 选择图片
2. 选择 type
3. 选择品种
4. 选择周期
5. 选择时间（默认当前时间，可编辑）
6. 选择方向
7. 选择结果
8. 输入一句话备注
9. 保存

### 设计原则
- 以选择器/标签按钮为主，尽量少打字
- 一屏内完成核心录入
- 默认值尽可能智能

### 重点交互
- type 支持最近使用
- 品种支持最近使用
- 时间默认当前时间
- 保存后可立即进入“继续录入”或“刷刚保存的 type”

---

## 8.3 刷流页

### 页面目标
以极低摩擦的方式回看图片样本

### 页面结构
- 主体：全屏图片
- 底部悬浮信息层：
  - type
  - 品种
  - 周期
  - 时间
  - 结果
  - 一句话备注
- 右侧浮动操作：
  - 收藏
  - 编辑
  - 删除
  - 分享

### 交互逻辑
- 上滑：下一条
- 下滑：上一条
- 左滑：切换到下一个筛选流（后续）
- 点击：展开完整详情
- 长按：快速修改标签

### 体验要求
- 刷图必须流畅
- 切换时尽量有轻微动画
- 页面视觉上应让图片占主导地位

---

## 8.4 筛选页

### 页面目标
让用户快速进入一个“专题样本流”

### 支持的筛选条件
- type（单选/多选）
- 品种（单选/多选）
- 周期
- 时间范围
- 方向
- 结果
- 是否收藏

### 结果展示
点击“开始刷”后，进入对应筛选条件下的流页面。

### 额外能力
支持将筛选条件保存为“常用视图”，例如：
- 只看 RB + 30m + 摸顶难
- 只看 FG + 亏损单
- 只看本周样本

## 10. 用户流程

## 10.1 核心流程一：快速记录
1. 用户打开 App
2. 点击“导入图片”
3. 选择截图
4. 填写最少字段（type + 其他选填）
5. 保存成功
6. 回到首页 / 继续录入 / 直接刷刚刚的 type

## 10.2 核心流程二：按 type 刷流
1. 用户打开首页
2. 点击某个常用 type
3. 进入该 type 的流页面
4. 上下滑连续回看历史样本
5. 长按/点击可快速修改标签和备注

## 10.3 核心流程三：按条件复盘
1. 用户进入筛选页
2. 选择品种 + 周期 + 时间 + 结果
3. 点击“开始刷”
4. 浏览对应样本流

## 11. MVP 边界控制

为了让 AI 编码更稳，第一版应严格收敛，不要一开始做太多。

### 第一版必须做
- 本地图片导入
- 本地数据库存储
- 创建/编辑/删除笔记
- 标签系统
- 按条件筛选
- 流式浏览
- 收藏
- 最近使用记录

### 第一版不要做
- 云同步
- 用户系统
- 社区
- 多人共享
- AI 自动识别
- 图表分析
- 自动生成交易结论
- 复杂统计报表

原因：MVP 先验证“图片 + 标签 + 刷流”这个核心闭环是否成立。

## 12. 技术实现建议（给 AI 编程工具参考）

### 12.1 目标平台
优先移动端：
- Flutter（推荐）

原因：
- 跨平台
- UI 做图片流和卡片交互方便
- 适合快速做 MVP

### 12.2 本地存储
推荐：
- SQLite / Drift / Isar

### 12.3 图片存储
- 先存本地文件路径
- 不做云存储

### 12.4 页面导航
推荐：
- go_router

### 12.5 状态管理
推荐：
- Riverpod

### 12.6 架构建议
- feature-first 目录结构
- clean-ish architecture，但不要过度设计
- 先保证可迭代性和生成稳定性

### 12.7 AI 编码时的开发原则
- 优先写可运行 MVP
- 避免一开始上复杂动画和复杂状态
- 先保证录入、筛选、刷流三条主链路稳定
- 所有字段和筛选条件从第一版就尽量结构化

## 13. UI 风格建议

### 13.1 关键词
- 深色主题优先
- 图片内容优先
- 极简
- 有沉浸式刷流感
- 接近内容消费产品，而不是办公软件

### 13.2 视觉原则
- 图片尽量大
- 字段尽量精简
- 底部悬浮信息层半透明
- 交互按钮不喧宾夺主

### 13.3 风格参考
- 抖音/小红书的信息流浏览逻辑
- 卡片式标签选择
- 避免 Notion 那种文档感

---

## 14. 成功指标

MVP 上线后，可观察以下指标：

### 14.1 记录效率
- 平均创建一条图片笔记耗时 <= 10 秒

### 14.2 回看频率
- 用户每天刷样本的次数
- 每次平均浏览的样本数

### 14.3 样本积累
- 单用户 7 日内新增样本数
- 单用户收藏样本数

### 14.4 留存指标
- 用户是否持续复盘
- 是否持续使用筛选流

## 17. 结论

这不是一个传统笔记 App。

它的本质是：

**面向交易复盘场景的图片案例流工具。**

它的价值不在于帮助用户写更多，而在于帮助用户：

- 更快记录样本
- 更轻回看样本
- 更频繁刷样本
- 最终形成模式识别能力

最终目标不是“记笔记”，而是“训练判断”。

> 一张图，一条样本；
> 千张图，一套系统。
